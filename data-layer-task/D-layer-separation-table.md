## Practical separation rule

Use this rule when describing the system:

- If it is a conceptual vocabulary term like `Claim`, `RelationDef`, or `dgb:source`, it belongs primarily to the schema layer.
- If it is a persisted row, join table, JSON file, frontmatter field, block-prop structure, or RPC mapping input, it belongs to the data layer.
- If it is an API payload, RID, JSON-LD object, or Turtle file used for interchange, it belongs to the transport layer.

## Most common confusions in this repo

- Treating `dg_base.ttl` and `dg_core.ttl` as if they were the live DB model.
- Treating Next.js API payloads as the canonical internal representation.
- Treating Obsidian local fields like `publishedToGroupId` or `importedFromRid` as if they were normalized shared persistence.
- Treating `Concept` as only "nodes" when it actually stores node schemas, relation schemas, node instances, and relation instances.
